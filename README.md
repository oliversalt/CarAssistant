# CarPi — Raspberry Pi Car Assistant

A voice-activated AI assistant designed to run in a car on a Raspberry Pi 4B.
You speak, it listens, thinks, and talks back.

---

## Two Pipelines

There are two working pipelines. Both support web search and conversation history, but they differ in speed and which AI models they use.

---

### `gpt_pipeline.py` — Faster (recommended)

```
Mic → PyAudio → gpt-audio-mini → ElevenLabs / GPT audio → Speaker
```

This pipeline sends your recorded audio **directly** to OpenAI's `gpt-audio-mini-2025-12-15` model. The model handles both transcription and the response in a single API call — there is no separate Whisper step. This makes it noticeably faster.

**Two output modes:**

```bash
# ElevenLabs TTS — George voice, higher quality (default)
python gpt_pipeline.py

# GPT native audio output — one fewer API call, slightly faster
python gpt_pipeline.py --voice gpt
```

**How it works:**
1. Records 5 seconds of audio via PyAudio
2. Encodes the WAV file as base64 and sends it to `gpt-audio-mini-2025-12-15`
3. If the model decides a web search is needed, it calls the `web_search` tool (DuckDuckGo, no API key required) and loops back to get a final answer
4. The response is either spoken by ElevenLabs or played back as audio generated by GPT itself
5. The conversation history is kept in memory so follow-up questions work naturally

---

### `main.py` — Slower (Claude-based)

```
Mic → PyAudio → Whisper STT → Claude Haiku → ElevenLabs → Speaker
```

This pipeline breaks the process into three separate API calls: Whisper transcribes the audio, Claude generates a response, and ElevenLabs speaks it. Because each step is a separate round trip, it is slower than `gpt_pipeline.py`.

The advantage is that it uses **Claude Haiku** as the AI brain, which you may prefer for certain kinds of conversation or reasoning.

```bash
python main.py
```

**How it works:**
1. Records 5 seconds of audio via PyAudio
2. Sends the WAV to OpenAI Whisper (`whisper-1`) to get a text transcript
3. Sends the transcript to Claude Haiku (`claude-haiku-4-5`) with the full conversation history
4. If Claude decides a web search is needed it calls the `web_search` tool and loops back
5. ElevenLabs speaks the response

---

## Why neither pipeline uses the OpenAI Responses API

OpenAI's Responses API has a `previous_response_id` feature that stores conversation state server-side, meaning you only send the new message each turn instead of the full history. This would be ideal for a car assistant.

However, **the audio input models do not support the Responses API** — it only works with standard text/vision models. Because both pipelines rely on audio input (either directly via `gpt-audio-mini` or indirectly via Whisper), the full conversation history has to be sent with every request. History is trimmed to the last 20 exchanges to keep request sizes manageable.

## Setup

```bash
# Activate venv
venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt
```

Create a `.env` file with:
```
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
ELEVENLABS_API_KEY=
```

---

## Hardware (target deployment)

- Raspberry Pi 4B 4GB running Raspberry Pi OS Lite
- AB13X USB mic/speaker combo (`MIC_INDEX = 1` on PC dev machine)
- Deployed via GitHub, autostarts via systemd on boot
- SSH access via Tailscale (`salt-raspberry-pi`)
